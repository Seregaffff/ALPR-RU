Цель практики: Освоить полный цикл разработки прикладной системы компьютерного зрения: от постановки задачи и подготовки данных до обучения нескольких архитектур, экспериментального сравнения качества, анализа ошибок и внедрения лучшей модели в демонстрационное приложение с визуализацией результатов. Этапы выполнения:
1. Ознакомление с вариантом задания. Допускается выбор собственного варианта только по согласованию с руководителем практики. Подбор и описание данных для выбранной прикладной задачи. Необходимо указать тип разметки, количество классов, объём выборки и ограничения использования данных.
2. Подготовка данных: разбиение на train/validation/test, проверка разметки, нормализация, аугментация и подготовка структуры проекта.
3. Выбор архитектур нейронных сетей. Для выбранной задачи необходимо обучить или дообучить минимум 5 различных архитектур и сравнить их между собой.
4. Проведение экспериментального сравнения. Для каждой архитектуры необходимо зафиксировать метрики качества, скорость инференса, размер модели и основные ошибки.
5. Интеграция лучшей модели в демонстрационный модуль. В программе должны отображаться исходные данные, результат работы модели, уверенность предсказания и краткая статистика.
6. Вывод статистики. Реализовать сохранение истории запусков или результатов в JSON/БД, а также генерацию краткого отчёта в PDF или Excel.
7. Подготовка итогового отчёта с анализом результатов, таблицей сравнения архитектур и примерами успешной и ошибочной работы модели.
8. Обязательные требования к сравнению архитектур
В каждом варианте студент обязан обучить или дообучить не менее пяти архитектур нейронных сетей. Простая замена размера одной и той же модели, например только YOLOv8n, YOLOv8s, YOLOv8m, не считается полноценным сравнением пяти архитектур. Необходимо сравнивать разные архитектурные подходы или разные семейства моделей, подходящие для выбранного типа задачи. Для каждой архитектуры необходимо указать: схему обучения, размер входного изображения, число эпох, основные параметры обучения, метрики качества, среднее время обработки одного изображения или кадра, размер модели, а также краткий вывод о сильных и слабых сторонах модели.
Требования к оценке качества
1) Для классификации использовать accuracy, precision, recall, F1-score и confusion matrix.
2) Для детекции использовать mAP, precision, recall, количество ложных срабатываний и пропусков.
3) Для сегментации использовать IoU, Dice/F1, pixel accuracy и визуальный анализ масок.
4) Для трекинга использовать IDF1, MOTA/HOTA при наличии разметки, а также число потерь ID и визуальный анализ траекторий.
5) Для видеоаналитики дополнительно оценивать задержку обработки, стабильность результата по кадрам и качество работы на сложных сценах.

Тема: Детекция и распознавание автомобильных номеров Что сделать: создать модуль, который находит номерной знак на изображении автомобиля и распознаёт символы. Нужно разделить задачу на этапы: детекция номера, выравнивание области номера, распознавание символов и анализ ошибок. Необходимо подготовить данные, выполнить обучение или дообучение не менее 5 архитектур, сравнить модели по метрикам качества и скорости обработки, выбрать лучшую модель и объяснить причины выбора. Что должно получиться: прототип ALPR-системы: изображение автомобиля на входе, рамка номера и распознанная строка на выходе.


1. Теоретическое обоснование, постановка задачи и подготовка проекта
1.1 Постановка задачи

В рамках данного проекта решается задача автоматической детекции государственных регистрационных знаков транспортных средств на изображениях, полученных из реальной среды. Данная задача относится к области компьютерного зрения и классифицируется как задача object detection, целью которой является одновременное определение:

наличия объекта на изображении;
его пространственного положения;
координат ограничивающего прямоугольника (bounding box).

Основной практической целью разработки является создание модели, способной работать в составе системы автоматического распознавания автомобильных номеров (ALPR — Automatic License Plate Recognition).

1.2 Обоснование выбора архитектуры YOLOv8n

В качестве базовой архитектуры была выбрана модель YOLOv8n (You Only Look Once, version 8 nano), представляющая собой облегчённую версию семейства одноэтапных детекторов объектов.

Выбор данной архитектуры обусловлен следующими факторами:

1) Высокая скорость обработки

YOLOv8n предназначена для работы в режиме реального времени, что критически важно для задач видеопотока.

2) Низкие требования к вычислительным ресурсам

Модель успешно функционирует на GPU уровня NVIDIA GTX 1070 Ti, что делает её доступной для учебных и исследовательских проектов.

3) Высокая точность при малом размере модели

Несмотря на компактность, YOLOv8n демонстрирует конкурентоспособные метрики качества по сравнению с более тяжёлыми архитектурами.

4) Поддержка готовой экосистемы Ultralytics

Библиотека предоставляет:

удобный API обучения;
автоматическую валидацию;
визуализацию метрик;
экспорт моделей.

1.4 Подготовка и структура датасета

Для обучения модели использовался специализированный датасет изображений транспортных средств с аннотированными государственными регистрационными знаками.

Структура датасета:
train/ — обучающая выборка;
val/ — валидационная выборка;
test/ — тестовая выборка.

Каждое изображение сопровождается аннотацией в формате YOLO:

<class_id> <x_center> <y_center> <width> <height>

где координаты нормализованы относительно размеров изображения.

Особенности датасета:
все изображения содержат один основной класс: license-plate;
данные включают различные условия:
дневное и ночное освещение;
разные углы обзора;
различное качество изображений;
присутствуют реальные сцены дорожного движения.
